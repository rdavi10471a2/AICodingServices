# Architecture Diagramming\n\nUse this card when Claude/codex needs to turn ambiguous architecture, workflow, validation, indexing, MCP/tool, or agent behavior into a clear diagram.\n\nPrefer ASCII timelines first. Use Mermaid or another format only when the target doc already uses it or the operator asks.\n\n## First Move\n\nIf prose is slippery, force it onto these axes:\n\n- **Time:** what happens first, next, last.\n- **Actor:** operator, Claude, MCP adapter, shared service, compiler, database, UI.\n- **Artifact:** watched source, Working candidate, staged record, overlay, index row, source map, build output.\n- **Gate:** hash check, validation, launch, human review, build, index refresh, policy decision.\n- **Authority:** advisory, predictive, authoritative, durable, review evidence.\n- **Failure path:** noisy, rejected, stale, abandoned, forced, replan.\n\nIf two ideas sound similar, put them in different boxes and label the difference directly.\n\n## Use The MCP Surface\n\nBefore drawing CodingServices behavior, use MCP evidence where available:\n\n```text\nget_tool_manifest\nget_monitor_status\nget_workflow_status\nget_current_task_context\nget_solution_index_status\nget_solution_index_tree\nquery_solution_index\nfind_indexed_symbols\nfind_indexed_references\nfind_indexed_callers\nfind_indexed_relationships\nget_source_map\nlist_session_staged_records\nget_staged_record\ncompare_file\nget_staging_guide\n```\n\nUse tool descriptions as the live contract for argument names. Do not infer MCP behavior from CLI command names.\n\nWhen a host exposes capability-gated tool groups, prefer read-only exploration tools for documentation and diagramming. Do not require mutation tools to be visible. If mutation tools are hidden, label that as expected capability gating instead of a missing documentation signal.\n\n## Freshness Check\n\nBefore diagramming a repo with active branches or review docs, separate evidence by freshness:\n\n- **Implemented:** current code path or tests prove it.\n- **Published contract:** docs on the requested branch say it, but code may lag.\n- **Branch/proposal:** review notes, diagrams, PR docs, or findings describe intended behavior.\n- **Inference:** reasoned from source shape but not directly proven.\n\nLabel the diagram or key with the evidence class. If docs and code disagree, diagram the disagreement instead of smoothing it over.\n\n## Default Timeline Shape\n\n```text\nPLAN     declare intent / scope\n           |\nEDIT     create candidate artifacts\n           |\nGATE 1   predictive check\n           |\nREVIEW   human/tool review boundary\n           |\nGATE 2   authoritative confirmation\n           |\nSTATE    durable result / follow-up\n```\n\nAfter the diagram, add a short key explaining what the diagram proves.\n\n## Good Diagram Labels\n\n- `semantic overlay compile`\n- `HASH-ONLY integrity check`\n- `full build on real watched tree`\n- `Working candidate`\n- `staged immutable record`\n- `operator saves watched source here`\n- `index refresh closure`\n- `source-map evidence`\n\nAvoid mushy labels such as `validation`, `refresh`, `sync`, `process`, or `finalize` unless the next sentence defines them.\n\n## Documentation Engine Notes\n\nWhen this card is used as part of CodingServices documentation planning, prefer folder-local docs:\n\n```text\nSelectedFolder/\n  Docs/\n    Folder.aim.md\n    SomeClass.aim.md\n    manifest.aim.json\n```\n\nFor each `.cs` file doc, make a best effort to explain "who uses me" from indexed callers/references, source search, or explicitly weak inference. Do not imply a consumer relationship is semantic when the evidence is only textual.\n\nKeep generated docs compact:\n\n- folder docs describe folder purpose, owned concepts, dataflow, and evidence gaps;\n- file docs describe one source file's purpose, ownership, dataflow, key methods, known consumers, invariants, and evidence;\n- user-editable sections may come later, but generated sections must remain hash/freshness aware.\n\n## Portable Host Vocabulary\n\nKeep behavior host-neutral first:\n\n```text\nstage immutable review evidence\nlaunch human diff review\nrecord accepted/rejected decision with staged hash\n```\n\nThen add host names only when needed:\n\n```text\nMCP: stage_candidate_for_review -> launch_staged_diff -> record_diff_decision\nCLI: edit stage -> edit launch-diff -> edit record-decision\n```\n\nDo not mix MCP and CLI names inside ordinary operator steps.\n\n## CodingServices Example\n\n```text\nPLAN     start_monitor_session(filesPlanned=[A,B,C])\n           |\nEDIT     refresh/new + create Working candidates A', B', C'\n           |\nGATE 1   semantic overlay compile\n         watched tree + A' + B' + C'\n         predictive; may be noisy for Razor/generated artifacts\n           |\nSTAGE    stage_candidate_for_review(A,B,C)\n           |\nREVIEW   launch_staged_diff\n         operator merges staged bytes in WinMerge\n           |\nDECIDE   record_diff_decision\n         watched source changes here on accepted save\n           |\nGATE 2   full build on real watched tree\n         authoritative confirmation\n           |\nINDEX    rebuild affected index rows\n```\n\nKey read:\n\n- GATE 1 predicts whether the planned overlay should work.\n- GATE 2 confirms what actually landed.\n- If GATE 1 and GATE 2 disagree, the question is fidelity, placement, or merge drift.\n- The point where watched source mutates must be visible in the diagram.\n
