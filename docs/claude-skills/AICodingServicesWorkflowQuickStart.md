# CodingServices Workflow Quick Start\n\nUse this card first when Claude/Codex is editing watched source through CodingServices.\n\n## Binding\n\nClaude Code should bind to the CodingServices bridge entry, not the raw server directly. The bridge sends Claude's MCP traffic through the CodexUI-owned MCP hub so the live app records request/response telemetry.\n\nUse the CodexUI-owned bridge registration for the active workspace. When the bridge is started correctly, Codex should connect through server `aicodingservices`.\n\n```text\nserver: aicodingservices\n```\n\n## First Calls\n\n```text\nget_monitor_status\nget_workflow_status\nget_self_check\nget_staging_guide\nget_tool_manifest\n```\n\nUse the Solution Index before loading bodies:\n\n```text\nget_solution_index_status\nquery_solution_index\nfind_indexed_symbols\nget_indexed_symbol\nfind_indexed_references\nfind_indexed_callers\nfind_indexed_relationships\n```\n\nUse source-map tools before symbol edits. They are important, not legacy:\n\n```text\nget_source_map(scope: "file", mode: "selector")\nget_symbol(symbolSelectorJson)\nsubmit_symbol(path, symbolSelectorJson, replacement)\n```\n\nThe index is a broad discovery surface. Source maps and symbols are the precise edit surface for C# member/type surgery. Before changing, removing, renaming, moving, or changing the signature/visibility of any symbol, perform a blast-radius check with indexed references/callers/relationships plus one cross-check signal. Declare every affected watched file in `start_monitor_session` before editing. Pre-merge validation is the guardrail for missed impact, not the discovery step.\n\n## Session Rule\n\n- For MCP workflow mutations, `sessionId` is not optional in practice. Pass the same `sessionId` to `new_file`, `refresh_file`, every mutation tool, and `stage_candidate_for_review`.\n- If a file is not already indexed to exactly one owning project, include explicit `owningProjectPath` in `start_monitor_session(filesPlanned: ...)`. This commonly applies to new files, docs, root instruction files, and other non-indexed assets.\n- If `start_monitor_session` succeeds but a later mutation or stage call returns a generic tool error, first verify that the file is in the session plan and that the call includes the same `sessionId`.\n\n## Existing File Edit\n\n```text\nstart_monitor_session(filesPlanned: [...]) with the planned watched file set, even for one-file edits\nrefresh_file(sourceFilePath, sessionId)\nedit only the returned Working candidate using MCP tools, passing the same sessionId to every mutation tool\nstage_candidate_for_review(path, sessionId)\nlaunch_staged_diff(stagedRecordId) for every planned staged record before recording decisions\noperator reviews/saves in WinMerge\nrecord_diff_decision(stagedRecordId, "accepted", expectedStagedHash)\nrefresh_file before another edit to the same watched file\nask the user whether to run `dotnet build` for the live watched solution confirmed by the app or `get_monitor_status`\n```\n\nSafe editing tools include:\n\n- `replace_text_in_file` with `expectedMatches`.\n- `find_text_span` then `replace_span_in_file`.\n- `get_source_map`, `get_symbol`, then `submit_symbol` for precise C# replacements.\n- `add_symbol`, `remove_symbol`.\n- `add_using`, `remove_using`, `set_type_partial`.\n- `add_field`, `add_property`, `add_method`, `add_constructor`, `add_nested_type`.\n- `submit_file` only for new files, generated files, or deliberate whole-file replacement.\n\nCSS, JSON, config, markup, and other non-C# text assets use the same Working candidate, staging, WinMerge diff, and decision flow. They do not need semantic index rows to be diffable.\n\nDo not edit watched source directly. Do not edit staged runtime files. After staging, further candidate changes must go back through the Working file and be staged again.\n\n## New File Edit\n\n```text\nstart_monitor_session(filesPlanned: [...]) with the planned watched file set\nnew_file(sourceFilePath, sessionId)\nsubmit_file(path, content, sessionId)\nstage_candidate_for_review(path, sessionId)\nlaunch_staged_diff(stagedRecordId) for every planned staged record before recording decisions\noperator creates/saves watched file in WinMerge\nrecord_diff_decision(stagedRecordId, "accepted", expectedStagedHash)\nask the user whether to run `dotnet build` for the live watched solution confirmed by the app or `get_monitor_status`\n```\n\nRejected new-file decisions leave watched source absent.\n\nFor new files and non-indexed docs, do not rely on index ownership discovery. Put `owningProjectPath` in the planned file entries up front.\n\n## Validation And Review\n\n`launch_staged_diff` always runs pre-merge validation before opening the configured review surface. Browser staged review is the default; WinMerge remains available as an explicit fallback. If validation fails on an interactive Windows desktop, CodingServices shows `Yes Launch` and `Cancel`.\n\n- `Cancel`: WinMerge does not open; fix the Working candidate and stage again.\n- `Yes Launch`: the configured review surface opens despite failed validation; only record `accepted` if the operator deliberately accepted the candidate into watched source.\n- No dialog available: ask the operator in chat before using `forceValidation`.\n\nFor planned sessions, `launch_staged_diff` confirms staged overlay readiness before opening review. Launch/review every planned staged record before recording decisions. The full planned staged overlay build runs at the terminal planned accepted decision before the accepted set is treated as final and before post-accept index refresh proceeds.\n\nAccepted or accepted-normalized decisions return `indexRefresh`. In planned sessions, early accepts can defer refresh until the terminal accepted decision. Check that status before relying on fresh index rows.\n
